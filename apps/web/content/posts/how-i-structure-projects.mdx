People ask about my project structure because it looks over-engineered for a personal site. It is. That's the point.

My monorepo has 16 packages. Each external integration is its own package with its own types, tests, and CI. Here's why, and how it works.

## The Core Philosophy

**Boundaries force clarity.** When you can't just `import { something } from '../utils'`, you have to think about what the boundary actually is. What does this module *do*? What does it *own*?

My site isn't a monorepo because I need scale. It's a monorepo because boundaries make the code honest.

## The Package Layout

```
packages/
├── analytics/          # PostHog + Vercel Analytics
├── database/           # Neon/PostgreSQL client
├── design-system/      # shadcn/ui source of truth (30+ components)
├── email/              # Contact notification templates
├── github/             # GitHub API client (contribution graph, stats)
├── next-config/        # Shared Next.js config presets
├── observability/      # Sentry + Logtail
├── rate-limit/         # Upstash Redis / Vercel KV
├── security/           # Arcjet + Nosecone middleware
├── seo/                # Metadata, viewport, JSON-LD helpers
├── spotify/            # Spotify API (now playing, top tracks)
├── strava/             # Strava API (activities, runs)
├── testing/            # Shared test setup (mocks for ResizeObserver, etc.)
├── typescript-config/  # Shared tsconfig presets
├── wakatime/           # WakaTime API (coding hours)
└── x/                  # X/Twitter API (latest post)
```

Each package exports a `keys()` factory for env validation, composed in `apps/web/src/config/env.ts` via `@t3-oss/env-nextjs`.

## Why Not One Big Package?

I tried that. It works until:
- You want to test the GitHub client without loading Spotify
- You need different dependency versions
- A security audit flags a transitive dep in `spotify/` that pollutes `database/`
- You want to extract one package to another project

With boundaries, I can:
- Update `strava` without touching `github`
- Run `pnpm test --filter github` in CI
- Publish `@gabfon/github` independently (if I wanted to)
- Pin exact deps per package via `pnpm overrides`

## Composition Over Inheritance

Every package is standalone. The web app *composes* them:

```typescript
// apps/web/src/config/env.ts
import { createEnv } from '@t3-oss/env-nextjs';
import { keys as analyticsKeys } from '@gabfon/analytics';
import { keys as databaseKeys } from '@gabfon/database';
// ... etc

export const env = createEnv({
  extends: [
    analyticsKeys(),
    databaseKeys(),
    githubKeys(),
    // all other packages
  ],
  // ...
});
```

This means:
- Adding a new integration = create package, add to `extends`
- No env var bleeding between packages
- Types flow through automatically

## Content as Code

Content lives in `content/posts/` and `content/projects/` as MDX. Fumadocs compiles it at build time to typed collections in `.source/`.

```typescript
// lib/content-index.ts
import { getPosts, getPostEntry } from '@/lib/content-index';

// Fully typed
const posts = getPosts(); // PostEntry[]
const post = getPostEntry('building-frontal'); // PostEntry | undefined
```

No database. No CMS. Just files that become typed React components.

## Server-First, Client-Only When Needed

The only `'use client'` components:
- `ThemeToggle` (needs localStorage)
- `ContactForm` (needs form state)
- `ContributionGraph` (needs canvas/interaction)
- `QueryProvider` (TanStack Query context)

Everything else is a Server Component. Data fetching happens in the component that needs it, passed down to presentational children.

## What I'd Change

- Fewer shadcn/ui components I don't use (I've started pruning)
- Extract the MDX components to a package (currently in `components/mdx/`)
- Better package READMEs with usage examples

## The Real Benefit

This structure forces me to answer: "What does this code *actually* depend on?"

When the answer is "everything," I know the design is wrong. The monorepo makes that visible immediately.

For a solo project, it's overkill. And that's exactly why it's valuable—it's practice for when the stakes are real.
